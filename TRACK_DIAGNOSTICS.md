# CAT062 航迹接收诊断指南

## 问题现象

程序运行了，但屏幕上没有看到航迹显示。

---

**在终端运行**:
```bash
cd /home/share/ATC_Display

```

这会输出详细的诊断信息，帮助定位问题。

### 问题情况 1: 没有收到 UDP 报文

**日志示例**:
```
2026-04-13 09:48:04,243 [INFO] atc_display.udp: CAT062 接收器已启动
2026-04-13 09:48:04,243 [INFO] atc_display: 窗口已显示, 使用 CLW 的 Quit 按钮退出
(长时间没有 UDP 轮询日志)
```

**可能原因**:
1. ❌ UDP 组播源没有发送数据
2. ❌ 组播地址或端口配置错误
3. ❌ 网络防火墙阻止
4. ❌ 网卡没有加入组播组

**解决方案**:
```bash
# 检查组播地址和端口配置
cat /home/share/ATC_Display/atc_display/config/ip_setting.json

# 检查网络连通性
tcpdump -i any udp port 8107  # 实时监听 8107 端口
netstat -ng                    # 检查加入的组播组
```

**正确的配置应为**:
```json
{
    "multicast_ip": "228.28.28.28",
    "multicast_port": 8107,
    "bind_host": "",
    "interface_ip": ""
}
```

### 问题情况 2: 收到报文但解析失败

**日志示例**:
```
2026-04-13 09:48:04,244 [DEBUG] atc_display.udp: 收到 CAT062 报文 #1, 大小: 100 字节
2026-04-13 09:48:04,245 [ERROR] atc_display.udp: 解析 CAT062 报文失败: 数据不足
```

**可能原因**:
1. ❌ 报文格式不符合 CAT062 标准
2. ❌ 报文被截断或损坏
3. ❌ parser 实现有 bug

**解决方案**:
- 检查 CAT062 报文格式是否正确
- 查看具体错误信息：`数据不足`、`无效的类别`等
- 检查 `cat062.py` 中的解析逻辑

---

### 问题情况 3: 报文解析成功但航迹为空

**日志示例**:
```
2026-04-13 09:48:04,244 [DEBUG] atc_display.udp: 收到 CAT062 报文 #1, 大小: 256 字节
2026-04-13 09:48:04,245 [DEBUG] atc_display.udp: 解析 CAT062 报文: 不包含航迹数据
```

**可能原因**:
1. ❌ CAT062 报文中没有航迹信息（可能是心跳包）
2. ⚠️ 航迹可能被过滤掉了（例如 SSR >= 7776 被过滤）

**解决方案**:
- 这是正常现象，不是所有报文都包含航迹
- 检查是否有 `新增 X 条航迹` 的日志

---

### 问题情况 4: 航迹被过滤

**日志示例**:
```
2026-04-13 09:48:04,245 [INFO] atc_display.udp: 解析 CAT062 报文成功: 包含 5 条航迹
2026-04-13 09:48:04,246 [DEBUG] atc_display.asd: 收到回调: 传入 5 条航迹数据
2026-04-13 09:48:04,246 [DEBUG] atc_display.asd: 更新 5 条现有航迹, 当前共 5 条航迹
```

**问题分析**:
- 解析了 5 条航迹，但全部是现有航迹的更新
- `新增 0 条`表示所有航迹都被过滤或更新了

**常见过滤条件**:
1. SSR >= 7776: 应答机编码无效
2. 坐标为 (0, 0)：位置信息无效
3. 超过 10 秒未更新：航迹超时

**检查方法**:
- 查看 `asd_widget.py` 中的 `TrackStore.update_tracks()`
- 查看 SSR 过滤条件是否过于严格

## 完整诊断步骤

### 第 1 步：确认接收器启动
```
2026-04-13 09:48:04,243 [INFO] atc_display.udp: CAT062 接收器已启动
```
✅ 看到这条日志 → 继续第 2 步

---

### 第 2 步：启用调试日志并运行
```bash
DEBUG=1 python -m atc_display 2>&1 | tee debug.log
```

等待 10 秒，观察是否有 UDP 轮询日志：
```
[DEBUG] atc_display.asd: UDP 轮询: 收到 X 个报文
```

✅ 有日志 → 继续第 3 步  
❌ 无日志 → 检查网络连接

---

### 第 3 步：检查报文是否被成功解析
查看是否出现：
```
[INFO] atc_display.udp: 解析 CAT062 报文成功: 包含 X 条航迹
```

✅ 有日志 → 继续第 4 步  
❌ 无日志 → 检查报文格式或 parser 实现

---

### 第 4 步：检查航迹是否被接收
查看是否出现：
```
[INFO] atc_display.asd: 新增 X 条航迹, 当前共 Y 条航迹
```

✅ 看到此日志 → 航迹已接收，检查渲染  
❌ 无此日志 → 检查过滤条件

---

### 第 5 步：检查航迹渲染
如果接收到航迹但屏幕没显示：
1. 检查航迹坐标是否在地图范围内
2. 检查地图中心坐标配置
3. 检查渲染代码是否有 bug

---
